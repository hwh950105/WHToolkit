[빌드/패키지 경고]
1. hwh/hwh/hwh.csproj는 OpenTK·SkiaSharp 패키지를 .NET Framework 타깃으로 복원해 NU1701 경고가 계속 발생합니다. net8.0/Windows를 공식 지원하는 최신 패키지로 교체하거나, hwh 프로젝트를 별도 솔루션으로 분리해 필요할 때만 빌드하도록 조정이 필요합니다.
2. WHToolkit 빌드 시 Database 헬퍼, OPC UA 클라이언트 등에서 nullability 경고(CS86xx)가 다수 발생합니다. 생성자 초기화/nullable 선언/가드 코드를 통해 정리하지 않으면 런타임 NRE 가능성이 남습니다.

[구조 및 의존성 이슈]
1. 기존 다중 프로젝트(WHToolkit.Core/Port/Socket/DeviceSamples)는 WHToolkit 단일 프로젝트에 병합됐지만, Porthelper/Sockethelper 폴더는 여전히 Compile Remove 처리돼 있습니다. 필요한 기능이라면 현 구조 안으로 완전히 통합하거나, 별도 아카이브로 분리해 혼란을 줄여야 합니다.
2. src/Core, src/Database_개발중, src/WHToolkit.Data 등 비활성 폴더가 그대로 남아 있습니다. 실제 사용 여부를 재평가해 삭제 또는 문서화가 필요합니다.
3. Porthelper/MachineBase.cs는 SampleDevice* 클래스를 기대하지만 DeviceSamples 폴더는 지금 빌드에 포함되지 않습니다. 실사용 계획이 있다면 새 통신 계층과 맞춰 구현을 복원해야 합니다.
4. Sockethelper/BaseObjects/SocketServer.cs 등 레거시 네트워크 코드가 Sockethelper.Models 등 미포함 폴더에 의존합니다. 재사용 의도가 있다면 Helper 계층으로 이전하거나 최신 네트워크 계층과 API를 통합해야 합니다.

[코드 품질/운영상 개선 사항]
1. NLog LogHelper는 Core 네임스페이스로 이동했지만 애플리케이션 시작점에서 Initialize/Shutdown을 반드시 호출하는 장치가 필요합니다.
2. DeviceSamples/SampleDevice* 클래스는 포트 이름/옵션을 직접 생성자에서 설정합니다. DI를 고려해 Options/Logger/RetryPolicy를 외부에서 주입받도록 리팩터링하면 테스트 가능성이 높아집니다.
3. Database Helper 계층은 DataParameter 등 public 타입에 XML 주석이 없어 CS1591 경고가 발생합니다. API 문서를 유지하려면 주석을 복원하거나 의미 있는 설명을 추가해야 합니다.
4. samples/, tests/ 폴더가 솔루션에 연결돼 있지 않습니다. 예제나 테스트 프로젝트를 활성화해 실제 DLL 사용 샘플과 검증 코드를 제공하면 배포 안정성에 도움이 됩니다.
5. .vs/, bin/, obj/ 등 빌드 산출물이 많이 남아 있으므로 .gitignore를 재검토하고 저장소를 정리하는 것이 좋습니다.
